# API Contract Sync

Verify that the TypeScript types in pfm-ui-react match the OpenAPI spec generated by pfm-go.
Produces a MATCH / DRIFT / MISSING report per endpoint schema. Optionally generates TypeScript
interfaces from the spec on user request.

Run this skill whenever:
- pfm-go handlers or routes change (after `/project:openapi-check`)
- A new API endpoint is added and frontend types need to be written
- You suspect frontend types have drifted from the backend spec

---

```

  Live:   https://pfm-go-api.zambone.dev/api/v1/openapi.yaml — unreachable
Run `make generate` in pfm-go to regenerate the local spec.
```

### 1.3 Report the source

```

```

## Phase 2 — Parse Endpoint Schemas

### 2.1 Extract all paths and methods

From the spec `paths` block, enumerate every `<path> → <method>` combination.

For each operation, extract:
- **Request schema** — the `$ref` under `parameters[?in=body].schema`, if present
- **Success response schema** — the `$ref` under `responses.200.schema` or `responses.201.schema`

Resolve each `$ref` to the corresponding entry in the `definitions` block. Collect the
resolved schema's `properties` map (field name → `type`).

### 2.2 Build the endpoint schema index

Produce an internal map:

```
<METHOD> <path>
  request:  { <field>: <json-type>, ... }  (empty if no request body)
  response: { <field>: <json-type>, ... }  (empty if no body e.g. 204)
```

JSON types come directly from the spec: `string`, `integer`, `boolean`, `number`, `array`,
`object`. Do not resolve nested `$ref`s — report the outer type only.

### 2.3 Report endpoint count

```
Parsed <N> endpoints from spec.
```

## Phase 3 — Locate TypeScript Types

### 3.1 Scan pfm-ui-react type files

Search these locations for TypeScript `interface` and `type` declarations:

```
/Users/fzambone/projects/pfm-ui-react/src/features/*/types.ts
/Users/fzambone/projects/pfm-ui-react/src/features/*/api/*.ts
```

For each file, extract all `interface <Name> { ... }` blocks. Collect field names and their
TypeScript types.

### 3.2 Build the TypeScript type index

Produce an internal map:

```
<InterfaceName>: { <field>: <ts-type>, ... }
```

### 3.3 Match spec schemas to TypeScript interfaces

For each endpoint schema from Phase 2, attempt to find a corresponding TypeScript interface.

**Matching heuristic (in order):**
1. Interface name that matches the spec schema name after stripping the `http.` prefix
   and converting to PascalCase (e.g., `http.loginResponse` → `LoginResponse`).
2. Interface name that contains the resource noun from the path and the HTTP method
   (e.g., `POST /auth/login` → look for `Login*` or `*Login`).
3. If no match is found → mark as **MISSING**.

## Phase 4 — Compare and Report

### 4.1 Type mapping

When comparing field types, use this mapping from JSON Schema → TypeScript:

| JSON Schema type | TypeScript equivalent |
|---|---|
| `string` | `string` |
| `integer` | `number` |
| `number` | `number` |
| `boolean` | `boolean` |
| `array` | `<T>[]` or `Array<T>` (any array variant) |
| `object` | `object` or any interface type |

A field with `type: integer` in the spec and `number` in TypeScript is a **MATCH**.

### 4.2 Per-endpoint verdict

For each endpoint in the index:

| Verdict | Condition |
|---|---|
| **MATCH** | Every spec field has a corresponding TypeScript field with a compatible type |
| **DRIFT** | At least one field name is missing from the TypeScript type, or has an incompatible type |
| **MISSING** | No TypeScript interface was found for this endpoint schema |

### 4.3 Report format

```
API Contract Sync — <date>
Spec source: <local|live>
Endpoints: <N total> | <M matched> | <D drift> | <X missing>

────────────────────────────────────────────────────────────
MATCH
────────────────────────────────────────────────────────────
POST /auth/login (response) → LoginResult
  token: string ✓
  expires_at → expiresAt: string ✓  [snake_case mapped]

────────────────────────────────────────────────────────────
DRIFT
────────────────────────────────────────────────────────────
GET /api/v1/households/{household_id}/accounts (response) → AccountResponse
  EXPECTED  id: string         ACTUAL  id: string         ✓
  EXPECTED  name: string       ACTUAL  name: string       ✓
  EXPECTED  balance: integer   ACTUAL  balance: string    ✗  (should be number)
  EXPECTED  version: integer   MISSING in TypeScript      ✗

────────────────────────────────────────────────────────────
MISSING — no TypeScript types found
────────────────────────────────────────────────────────────
POST /api/v1/households
GET  /api/v1/households/{household_id}
...
```

**Notes on snake_case mapping:**
- The backend uses `snake_case` JSON keys; the frontend conventionally maps to `camelCase`.
- A field `expires_at` in the spec matching `expiresAt` in TypeScript is a **MATCH** —
  note the mapping in the report but do not count it as drift.

## Phase 5 — Optional Type Generation

After the report, ask:

```
Generate TypeScript interfaces for MISSING or DRIFT endpoints? (y/n)
```

If **no**: stop here.

If **yes**: ask which endpoints to generate for (default: all MISSING + DRIFT).

For each selected endpoint schema, generate a TypeScript interface:

**Rules:**
- Convert field names from `snake_case` to `camelCase`
- Map JSON types using the table in Phase 4.1
- For `array` types, use `<element-type>[]` where element type is `unknown` if unresolved
- Add a JSDoc comment citing the spec path/method and schema name
- Output to `src/features/<resource>/types.ts` in pfm-ui-react, where `<resource>` is
  inferred from the path (e.g., `/api/v1/households/...` → `households`)

**Format:**
```typescript
/**
 * Auto-generated from OpenAPI spec.
 * Endpoint: <METHOD> <path>
 * Schema: <spec-schema-name>
 */
export interface <InterfaceName> {
  <camelCaseField>: <tsType>;
  ...
}
```

**Stop before writing.** Show the generated interfaces and ask:
```
Write these interfaces to pfm-ui-react? (y/n)
```

Only write on explicit confirmation.
